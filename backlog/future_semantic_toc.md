Evolve Synapse's Table of Contents system from a path-based file index into a semantic knowledge graph that understands code purpose, relationships, and change patterns. The current TOC (generated by `_commands/project/toc_generate.md` and maintained by `_commands/project/toc_update.md`) is essentially a structured `ls -R` with summaries — it knows file paths, one-line descriptions, and tags, but it does not understand semantic relationships between code elements. When a component's purpose changes (e.g., a utility becomes a service, a helper evolves into a core module), the TOC does not detect it unless the file is physically moved. The search command (`_commands/project/toc.md`) can only do keyword matching against summaries and tags — it cannot follow dependency chains, identify file clusters, or detect API changes.

This swarm has five concrete deliverables:

---

**1. Add semantic fingerprinting to TOC generation**

Modify `_commands/project/toc_generate.md` to produce semantic fingerprints alongside the existing TOC entries. Currently, scan agents in Phase 2 (Step 4-5, around line 56-101) report: file path, summary, tags, related files, and exports. Extend the agent prompt to also report a structured "semantic fingerprint" per file.

The fingerprint contains four fields:

a. **`purpose`** — One of a fixed set of categories: `component`, `service`, `utility`, `config`, `test`, `type-definition`, `route`, `middleware`, `hook`, `model`, `migration`, `script`, `documentation`, `command`, `style`, `entry-point`, `factory`, `context-provider`. The agent selects the single most accurate category based on the file's primary role. If none fit, use `other` with a parenthetical (e.g., `other (build plugin)`).

b. **`key_exports`** — Array of objects: `{ "name": "functionOrClassName", "kind": "function|class|type|interface|constant|enum", "params": 3 }`. Only the top 5-10 most important exports — not every helper. `params` is the parameter count for functions (omit for non-functions).

c. **`key_imports`** — Array of strings: relative paths to other project files this file imports from. Only project-internal imports — exclude node_modules and external packages. Example: `["../utils/json.js", "../services/DashboardService.js"]`.

d. **`complexity`** — One of `simple`, `moderate`, `complex`. Heuristic: `simple` = under 100 lines with linear logic, `moderate` = 100-300 lines or has branching/async patterns, `complex` = 300+ lines or has deeply nested logic, multiple state machines, or heavy algorithmic content.

Store fingerprints in a sidecar file at `{project_root}/.synapse/fingerprints.json` with this structure:

```json
{
  "generated_at": "2026-03-21",
  "project_root": "/path/to/project",
  "files": {
    "src/server/index.js": {
      "purpose": "entry-point",
      "key_exports": [
        { "name": "startServer", "kind": "function", "params": 1 }
      ],
      "key_imports": [
        "src/server/services/WatcherService.js",
        "src/server/services/DashboardService.js",
        "src/server/utils/json.js"
      ],
      "complexity": "moderate"
    },
    "src/ui/utils/dependencyLines.js": {
      "purpose": "utility",
      "key_exports": [
        { "name": "drawDependencyLines", "kind": "function", "params": 5 },
        { "name": "setupCardHoverEffects", "kind": "function", "params": 2 }
      ],
      "key_imports": [],
      "complexity": "complex"
    }
  }
}
```

Update the scan agent prompt template in Step 4 (around line 66-93) to request the fingerprint alongside existing fields. The agent's output format becomes:

```
### {filename}
- **Path:** `{relative_path}`
- **Summary:** {description}
- **Tags:** {tag1}, {tag2}, ...
- **Related:** `{path1}`, `{path2}`, ...
- **Exports:** `Symbol1`, `Symbol2`, ...
- **Fingerprint:**
  - Purpose: {purpose}
  - Key Exports: {name}({kind}, {params}), ...
  - Key Imports: `{path1}`, `{path2}`, ...
  - Complexity: {simple|moderate|complex}
```

Update the master's assembly logic in Phase 3 (Step 6-7, around line 105-175) to:
- Write fingerprint data to `{project_root}/.synapse/fingerprints.json` as a separate file alongside `toc.md`
- Do NOT embed fingerprint data inline in `toc.md` — the TOC remains human-readable markdown. Fingerprints are machine-readable JSON consumed by other Synapse commands.

Update Step 8 (reporting, around line 177-189) to include the fingerprint file in the report.

Files to modify:
- `_commands/project/toc_generate.md`

**2. Build a file-level dependency graph from imports**

During TOC generation, after all scan agents return fingerprints containing `key_imports`, the master assembles a complete file-level dependency graph. Store it at `{project_root}/.synapse/dep_graph.json`:

```json
{
  "generated_at": "2026-03-21",
  "project_root": "/path/to/project",
  "graph": {
    "src/server/index.js": {
      "imports": [
        "src/server/services/WatcherService.js",
        "src/server/services/DashboardService.js",
        "src/server/utils/json.js",
        "src/server/utils/constants.js"
      ],
      "imported_by": [
        "electron/main.js"
      ]
    },
    "src/server/utils/json.js": {
      "imports": [
        "src/server/utils/constants.js"
      ],
      "imported_by": [
        "src/server/index.js",
        "src/server/services/WatcherService.js",
        "src/server/services/DashboardService.js"
      ]
    }
  }
}
```

The `imports` field comes directly from each file's `key_imports` in the fingerprint data. The `imported_by` field is the reverse mapping — computed by the master after all fingerprints are collected. For every file A that lists file B in its `key_imports`, B's `imported_by` includes A.

This dependency graph enables three things:
- The `!scope` command can follow real import chains instead of heuristic-based file discovery
- The `!toc` search command can follow dependency paths (deliverable 4 below)
- The master agent can identify file clusters for better task decomposition during swarm planning (deliverable 5 below)

Update `_commands/project/toc_generate.md` Phase 3 to include dependency graph assembly alongside fingerprint assembly. The master reads all returned fingerprints, extracts `key_imports`, builds the forward map (`imports`), computes the reverse map (`imported_by`), and writes `dep_graph.json`.

Files to modify:
- `_commands/project/toc_generate.md`

**3. Detect purpose changes and API changes during TOC update**

Modify `_commands/project/toc_update.md` to compare old fingerprints against new fingerprints for modified files, flagging significant semantic changes.

Currently, `!toc_update` (in `_commands/project/toc_update.md`) computes a diff of new/deleted files (Phase 1, Steps 1-3, around line 10-68), dispatches agents to scan new files (Phase 2, Steps 4-6, around line 75-116), and applies changes (Phase 3, Steps 7-10, around line 120-171). It does NOT detect when existing files have changed semantically.

Add a new detection category between Steps 2 and 3: **Modified files**. Use git or file modification times to identify files that exist in both the TOC and on disk but have been modified since the last TOC generation. The `generated_at` field in `fingerprints.json` provides the reference date.

Detection approach:
- Read `{project_root}/.synapse/fingerprints.json` to get the `generated_at` date
- Use `git diff --name-only` with the date (or `find` with `-newer`) to identify files modified since generation
- Cross-reference with the fingerprints file — only files that have existing fingerprints AND have been modified need re-scanning

For modified files, dispatch scan agents (same as for new files) that produce updated fingerprints. Then compare old fingerprint vs new fingerprint for each modified file:

**Purpose change detection:**
- If `purpose` changed (e.g., `utility` to `service`), flag as: `"PURPOSE CHANGE: {path} changed from {old_purpose} to {new_purpose}"`
- This is a significant semantic shift that should be prominently reported

**API change detection:**
- Compare `key_exports` arrays. If more than 50% of export names differ (by name), flag as: `"API CHANGE: {path} — {N} of {M} exports changed ({list of removed}, {list of added})"`
- If export signatures changed (same name but different `kind` or significantly different `params`), flag as: `"SIGNATURE CHANGE: {path} — {export_name} changed from {old} to {new}"`

**Dependency change detection:**
- Compare `key_imports` arrays. If imports were added or removed, flag as: `"DEPENDENCY CHANGE: {path} — added imports from {list}, removed imports from {list}"`
- Update `dep_graph.json` with the new import relationships

Report format (added to the existing diff output in Step 3):

```markdown
### Modified Files (Semantic Changes)
- `src/utils/auth.js` — PURPOSE CHANGE: utility -> service
- `src/api/users.js` — API CHANGE: 3 of 5 exports changed (removed: getUser, listUsers; added: fetchUser, searchUsers, getUserById)
- `src/services/db.js` — DEPENDENCY CHANGE: added import from `src/utils/cache.js`
```

Update fingerprints.json and dep_graph.json with the new data for modified files.

Files to modify:
- `_commands/project/toc_update.md`

**4. Add semantic search commands to `!toc`**

Extend `_commands/project/toc.md` to support four new search modes alongside the existing keyword search. The command currently (Steps 1-4, around line 20-84) reads the TOC, matches against file names/summaries/tags/related files, verifies candidates, and reports results.

Add these new sub-commands:

**a. `!toc depends-on {file_path}`**
Show all files that directly import the specified file. Read `{project_root}/.synapse/dep_graph.json`, look up the file's `imported_by` array, and display each importing file with its TOC summary. If `dep_graph.json` doesn't exist, tell the user to run `!toc_generate` first.

Example:
```
!toc depends-on src/server/utils/json.js

## Depends On: src/server/utils/json.js

4 files import this module:

1. **WatcherService.js**
   - Path: `src/server/services/WatcherService.js`
   - Summary: Watches dashboard directories for file changes, broadcasts SSE events
   - Uses: readJSON, readJSONWithRetry, isValidInitialization, isValidProgress

2. **DashboardService.js**
   - Path: `src/server/services/DashboardService.js`
   - Summary: Dashboard CRUD operations and directory management
   - Uses: readJSON

...
```

**b. `!toc depended-by {file_path}`**
Show all files that the specified file imports from. Read `dep_graph.json`, look up the file's `imports` array, and display each dependency with its TOC summary. This is the reverse of `depends-on`.

**c. `!toc cluster {keyword}`**
Find all files related to a keyword by following dependency chains. Algorithm:
1. Find all files matching the keyword (by filename, summary, or tags — same as existing search)
2. For each matching file, follow its `imports` and `imported_by` one level deep (direct dependencies only — not transitive, to avoid exploding the result set)
3. Deduplicate and present the full cluster: the matching files plus their direct dependency neighborhood
4. Group results: "Core files" (matched the keyword directly) and "Related files" (connected via dependencies)

Example:
```
!toc cluster auth

## Cluster: "auth"

### Core Files (matched "auth")
- `src/server/middleware/auth.js` — JWT validation middleware
- `src/server/services/AuthService.js` — Authentication business logic
- `src/ui/hooks/useAuth.js` — React auth state hook

### Related Files (dependency neighbors)
- `src/server/utils/jwt.js` — imported by auth.js
- `src/server/routes/apiRoutes.js` — imports auth.js
- `src/ui/context/AppContext.jsx` — imports useAuth.js
```

**d. `!toc changes-since {git_ref}`**
Show which TOC entries have changed purpose or API since a given git reference (e.g., `HEAD~5`, `main`, a commit hash). Algorithm:
1. Run `git diff --name-only {git_ref}` to get the list of changed files
2. For each changed file that has a fingerprint in `fingerprints.json`, compare the current file against the stored fingerprint
3. Dispatch scan agents for the changed files to get current fingerprints (same agent prompt as `!toc_generate`)
4. Compare old vs new fingerprints and report purpose changes, API changes, and dependency changes (same detection logic as deliverable 3)
5. Do NOT modify `fingerprints.json` or `dep_graph.json` — this is a read-only query. Use `!toc_update` to persist changes.

The command resolution logic at the top of `_commands/project/toc.md` needs to detect which sub-command is being invoked. Parse the first argument after `!toc`:
- If it matches `depends-on` -> sub-command a
- If it matches `depended-by` -> sub-command b
- If it matches `cluster` -> sub-command c
- If it matches `changes-since` -> sub-command d
- Otherwise -> existing keyword search behavior (unchanged)

Files to modify:
- `_commands/project/toc.md`

In Step 6 "Decompose into tasks" (around line 98-115), add a "Coupling Check" after the decomposition is complete:

```
#### Coupling Check (when dep_graph.json is available)

After decomposing into tasks and assigning waves, verify that same-wave tasks do not
modify tightly-coupled files. For each pair of tasks in the same wave:

1. List all files each task modifies
2. Check dep_graph.json for direct import relationships between those files
3. If Task A modifies file X and Task B modifies file Y, and X imports Y (or vice versa):
   - This is a coupling risk. The two tasks may create conflicting changes.
   - Options: (a) make them sequential (add a dependency), (b) merge them into one task,
     or (c) keep them parallel but add explicit coordination notes in both task contexts
     warning about the coupling and specifying which task "owns" the shared interface
4. If both tasks modify files that import the same shared module:
   - Lower risk, but note it in the task context so workers are aware of the shared dependency

Document coupling decisions in the parallelization plan (Step 8) under a new
"## Coupling Analysis" section, listing each coupling pair found and the resolution chosen.
```
