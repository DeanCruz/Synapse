# 2.2 Swarms and Dashboards

A swarm is a group of agents working on separate tasks. The master agent plans and coordinates. Worker agents implement the tasks and report progress.

## Start A Tracked Swarm

Type a clear `!p_track` request:

```text
!p_track Build a notification system with database schema, API endpoints, frontend notification bell, dropdown panel, unread counts, and tests. Follow existing patterns.
```

Good swarm prompts include:

- The feature or outcome.
- Major parts of the work.
- Relevant files, modules, or patterns if you know them.
- Constraints, such as "do not change the auth model" or "keep existing API shapes".
- Test or verification expectations.

## Review The Plan

Before workers start, Synapse populates the dashboard with the planned tasks and dependency lines. Check:

- Task names are specific.
- File ownership does not overlap unexpectedly.
- Dependencies make sense.
- Independent work is in the same wave.
- Risky work has verification.

Approve the plan only when the task split matches what you want.

## What The Dashboard Shows

Agent cards move through these stages:

| Stage | Meaning |
|---|---|
| `reading_context` | The worker is reading instructions and relevant project files. |
| `planning` | The worker is choosing its implementation approach. |
| `implementing` | The worker is creating or modifying files. |
| `testing` | The worker is validating its changes. |
| `finalizing` | The worker is preparing its summary. |
| `completed` | The task finished successfully. |
| `failed` | The task failed and needs attention. |

Click an agent card to inspect milestones, logs, files changed, deviations, and errors.

The dashboard combines the original plan with each worker's live progress file, so it can show current status, elapsed time, files changed, logs, and deviations as workers report them.

## Dependencies

Some tasks depend on others. For example, a UI task may need an API task to finish first. Synapse dispatches all unblocked work immediately and starts dependent tasks as soon as their prerequisites complete.

Dependency lines show what blocks what. If a task is pending longer than expected, inspect its upstream dependencies.

## Deviations

A yellow deviation badge means a worker did something different from the original plan. This can be fine. Examples:

- The worker touched an extra file because the existing pattern required it.
- The worker skipped a planned file because the relevant code lived elsewhere.
- The worker changed approach while preserving the requested outcome.

Inspect deviations before accepting the final result.

## Failures And Retries

If a task fails:

1. Click the failed card or run `!inspect {task_id}`.
2. Read the error and final worker summary.
3. If the failure is isolated, run `!retry {task_id}`.
4. If several tasks fail together, let Synapse replan or ask for a new plan with the failure context.

Synapse can trigger a circuit breaker when failures threaten many downstream tasks. In that case, it diagnoses the root cause, creates repair work when appropriate, and resumes from a safer plan.

## After Completion

When all workers finish:

1. Read the final report.
2. Inspect deviations and failed or partial tasks.
3. Open Git Manager or your terminal to review the diff.
4. Run your project tests if workers did not already verify everything.
5. Commit only after you understand the changes.
