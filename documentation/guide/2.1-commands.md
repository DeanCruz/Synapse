# 2.1 Commands

Synapse commands start with `!`. You can type them in Chat or use the Commands browser in the app header to discover and run them.

## How Commands Are Found

When you type `!command`, Synapse finds a matching markdown command file and follows it as the source of truth. The Commands browser is a read-only view of the available command files, except for user-local commands that you intentionally create or edit.

Resolution order:

1. Synapse swarm commands.
2. Synapse project commands.
3. User-local commands.
4. Project-specific commands.

The first match wins. Command files are read fresh every time, so the behavior comes from the current command file rather than memory.

## Commands You Will Use Most

| Command | Use It When |
|---|---|
| `!p_track {prompt}` | You want a full tracked swarm with dashboard visibility. |
| `!p {prompt}` | You want quick parallel work with less tracking overhead. |
| `!status` | You want a progress summary for the active dashboard. |
| `!logs` | You want the event log. |
| `!inspect {task_id}` | You want details for one worker task. |
| `!retry {task_id}` | A task failed and should be rerun with failure context. |
| `!project` | You want to inspect or manage the current project path. |
| `!initialize` | You want to set up Synapse for a project. |
| `!learn` | You want to build the Project Knowledge Graph. |
| `!learn_update` | You want to refresh stale or new knowledge graph entries. |
| `!context {query}` | You want focused project context before deciding what to do. |
| `!review` | You want a code review of changes or files. |
| `!health` | You want a project health check. |
| `!instrument` | You want Live Preview inline editing support. |

Use the Guide button when you need app help. Use the Commands browser when you need the exact command list or command text.

## Choosing A Work Command

Use this rule:

| Situation | Command |
|---|---|
| One question or one small edit | Plain chat request |
| Two or more related edits that do not need a dashboard | `!p` |
| Three or more pieces, dependencies, or user-visible risk | `!p_track` |
| Several independent swarms across a large body of work | `!master_plan_track` |

## Monitoring Commands

During a tracked swarm:

```text
!status
!logs --last 20
!logs --level error
!inspect 2.3
!deps --blocked
```

The dashboard is the primary view, but these commands help when you need details in chat.

## Project Commands

Project commands help agents understand the target repo:

```text
!initialize
!scaffold
!learn
!learn_update
!context "authentication middleware"
!scope "adding a user role field"
!review
```

Run these before large work when the project is unfamiliar or recently changed.

## Profiles

Profiles adjust the agent's priorities. Put a profile before a command:

```text
!security !p_track Add input validation and rate limiting to all public endpoints.
!qa !review
!architect !plan Replace the queue system with a durable job backend.
```

Use profiles when the work needs a specific lens, such as security, QA, architecture, product, or technical writing.
