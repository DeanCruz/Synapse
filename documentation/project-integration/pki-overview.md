# Project Knowledge Index (PKI)

The Project Knowledge Index (PKI) is a **persistent, auto-accumulating knowledge layer** at `{project_root}/.synapse/knowledge/`. It gives future sessions deep operational understanding of the codebase -- gotchas, patterns, conventions, domain taxonomy, and file relationships -- without redundant exploration.

---

## Getting Started

Bootstrap the PKI for any target project in three steps:

1. **Point Synapse at your project:** `!project set /path/to/your/project`
2. **Run the bootstrap:** `!learn` -- dispatches a parallel swarm that reads every source file and produces structured annotations.
3. **Verify:** Check that `{project_root}/.synapse/knowledge/manifest.json` exists and `stats.annotated > 0`.

After bootstrapping, the PKI auto-maintains itself:
- Workers mark annotations stale when they modify files (via PostToolUse hook)
- Run `!learn_update` periodically to re-scan stale/new files
- Workers contribute new knowledge during normal swarm execution

```
│  REFRESH                 CONSUME                            │

│    re-scans stale only     injects into worker prompts      │

└─────────────────────────────────────────────────────────────┘
```

## Directory Structure

```
{project_root}/.synapse/knowledge/
├── manifest.json          ← Master routing index (domain_index, tag_index, concept_map)
├── annotations/           ← Per-file deep knowledge (hash-keyed)
│   ├── a1b2c3d4.json
│   └── ...
├── domains.json           ← Auto-discovered domain taxonomy
├── patterns.json          ← Cross-cutting patterns and conventions
└── queries/               ← Pre-computed domain bundles (gitignored)
```

| File | Purpose |
|---|---|
| `manifest.json` | Routes queries to annotations via `domain_index`, `tag_index`, `concept_map`. Tracks staleness per file. |
| `annotations/{hash}.json` | Per-file knowledge: purpose, exports, imports, gotchas, patterns, conventions, relationships. Hash = first 8 chars of SHA-256 of relative path. |
| `domains.json` | Domain taxonomy grouping files by functional area. |
| `patterns.json` | Cross-cutting patterns observed across the codebase. |
| `queries/` | Cached domain bundles. Gitignored -- regenerated by `!context`. |

Full schemas: [`documentation/data-architecture/pki-schemas.md`](../data-architecture/pki-schemas.md)

## Population Mechanisms

### 1. `!learn` -- Cold-Start Bootstrap

Dispatches a parallel swarm where workers deeply read files and produce structured annotations. The master assembles these into manifest, annotations, domains, and patterns files. Use for first-time PKI generation or after major refactors.

Command spec: [`_commands/project/learn.md`](../../_commands/project/learn.md)

### 2. Worker Annotations -- Swarm-Time Accumulation

Workers discover knowledge during normal execution and report it via the `annotations` field in their progress files. The master harvests these at swarm completion and merges them into the PKI.

Worker guidance: [`agent/instructions/tracker_worker_instructions.md`](../../agent/instructions/tracker_worker_instructions.md)

### 3. Staleness Hook -- Automatic Change Detection

The PostToolUse hook (`.claude/hooks/mark-annotation-stale.sh`) marks annotations stale when any agent writes to an annotated file. Stale annotations remain usable but signal that the file changed since last annotation.

### 4. `!learn_update` -- Incremental Refresh

Detects stale, new, and deleted files, then dispatches workers only for files needing re-annotation. Runs quickly enough for pre-swarm execution. Suggests `!learn` if 50+ files need scanning.

Command spec: [`_commands/project/learn_update.md`](../../_commands/project/learn_update.md)

## Planning Integration

### Master Pre-Planning (6 Steps)

When a master begins planning, it checks for PKI data and injects relevant knowledge into worker prompts. Full protocol: [`agent/master/pki_integration.md`](../../agent/master/pki_integration.md).

1. Check if `manifest.json` exists (skip gracefully if not)
2. Extract domains, tags, concepts from user prompt
3. Look up files via `domain_index`, `tag_index`, `concept_map`
4. Read annotations for matched files (max 8-10 reads)
5. Extract gotchas, patterns, conventions
6. Inject into worker prompts' CONVENTIONS section

**Key behaviors:**
- No PKI = no error, just standard planning
- Gotchas prioritized over patterns and conventions
- Each worker gets only PKI knowledge relevant to their specific files
- Stale annotations included with caveats

### Worker Prompt Injection

PKI data merges into existing CONVENTIONS and CONTEXT sections (no separate PKI section). Items are labeled with `[PKI]` markers. Priority when trimming: gotchas > patterns > relationships > conventions.

Details: [`agent/master/worker_prompts.md`](../../agent/master/worker_prompts.md)

### `!context` PKI Quick-Route

The `!context` command queries the PKI before falling back to grep/glob. When PKI data is available, output includes Known Gotchas and Discovered Patterns sections alongside standard results.

Details: [`_commands/project/context.md`](../../_commands/project/context.md)
