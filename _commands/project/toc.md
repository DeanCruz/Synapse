# `!toc {sub-command|query}`

**Purpose:** Search the project Table of Contents to quickly locate files by topic, keyword, or object name. Also supports dependency-aware queries via sub-commands when `dep_graph.json` and `fingerprints.json` exist. Returns the most relevant file(s) with their paths and context — faster than blind Glob/Grep searching.

**Syntax:** `!toc {sub-command} {argument}` or `!toc {search_query}`

**Sub-commands:**

| Sub-command | Syntax | Description |
|---|---|---|
| `depends-on` | `!toc depends-on {file_path}` | Show all files that import the specified file |
| `depended-by` | `!toc depended-by {file_path}` | Show all files the specified file imports from |
| `cluster` | `!toc cluster {keyword}` | Find files matching keyword + their dependency neighbors |
| `changes-since` | `!toc changes-since {git_ref}` | Report purpose/API/dependency changes since a git reference |

**Keyword search (default):** `!toc {search_object}` or `!toc {search_parameters}`

- `{search_object}` — A specific thing to find (e.g., `SimulationService`, `useExplore`, `auth middleware`, `landing page gallery`)
- `{search_parameters}` — Descriptive search (e.g., `rate limiting`, `styling colors`, `how generations work`)

**Examples:**
```
!toc SimulationService
!toc explore page hooks
!toc auth middleware
!toc how the generation flow works
!toc firebase config
!toc depends-on src/server/utils/json.js
!toc depended-by src/server/index.js
!toc cluster auth
!toc changes-since HEAD~5
```

**Tag-based filtering:**

- `!toc #tag` — Search only entries tagged with `{tag}`. The `#` prefix triggers tag-only mode.
- `!toc #tag1 #tag2` — Search entries tagged with ALL specified tags (AND logic).
- `!toc #tag query` — Search entries tagged with `{tag}` AND matching `{query}` in name/summary.

---

## Command Resolution

Parse the first argument after `!toc` to determine which mode to execute:

| First argument | Mode | Action |
|---|---|---|
| `depends-on` | Sub-command A | Jump to **Sub-command A: depends-on** |
| `depended-by` | Sub-command B | Jump to **Sub-command B: depended-by** |
| `cluster` | Sub-command C | Jump to **Sub-command C: cluster** |
| `changes-since` | Sub-command D | Jump to **Sub-command D: changes-since** |
| Anything else | Keyword search | Jump to **Keyword Search** (existing behavior, unchanged) |

The match is exact and case-sensitive. If the user types `!toc depends-on src/foo.js`, the first argument is `depends-on` and the remainder (`src/foo.js`) is the argument to the sub-command. If the user types `!toc dependency`, it does NOT match any sub-command — it falls through to keyword search for the term "dependency".

## Sub-command A: `!toc depends-on {file_path}`

**Purpose:** Show all files that directly import the specified file — i.e., the file's reverse dependencies ("who depends on me?").

**Requires:** `{project_root}/.synapse/dep_graph.json` (generated by `!toc_generate`)

### Step A1: Verify dep_graph.json exists

Check for `{project_root}/.synapse/dep_graph.json`. If it does not exist, report:

```
dep_graph.json not found at {project_root}/.synapse/dep_graph.json.

Run `!toc_generate` to produce the dependency graph alongside the TOC.
The dependency graph is built from import analysis during TOC generation.
```

Stop execution. Do not fall back to keyword search.

### Step A2: Read the dependency graph

Read `{project_root}/.synapse/dep_graph.json` in full. Parse the `graph` object.

### Step A3: Look up the file

Normalize the provided `{file_path}` to be relative to the project root (strip any leading `{project_root}/` or `./`).

Look up the normalized path in `graph`. If the file is not in the graph, report:

```
File `{file_path}` not found in the dependency graph.

Possible causes:
- The file may not have been scanned during TOC generation (check if it's excluded by skip rules)
- The file may have been created after the last `!toc_generate` run
- The path may be incorrect — verify the file exists on disk

Suggestion: Run `!toc_update` to re-scan changed files, or `!toc_generate` for a full rebuild.
```

### Step A4: Retrieve importers

Read the file's `imported_by` array from the graph. If the array is empty, report that no files import this module.

### Step A5: Enrich with TOC summaries

Read `{project_root}/.synapse/toc.md`. For each file in the `imported_by` list:
1. Find its entry in the TOC
2. Extract the summary, tags, and exports

If `fingerprints.json` exists, also read it and extract the importing file's `key_exports` to show which symbols it uses (cross-reference with the target file's `key_exports` to identify which exports are consumed).

### Step A6: Report results

```markdown
## Depends On: {file_path}

{N} file(s) import this module:

**1. {filename}**
- **Path:** `{relative_path}`
- **Summary:** {TOC summary}
- **Tags:** {tags from TOC}
- **Uses:** {list of exports from the target file that this importer likely consumes, if determinable from fingerprints}

**2. {filename}**
- ...

### Not in TOC
{List any importing files that appear in dep_graph.json but have no TOC entry — suggest `!toc_update`}
```

## Sub-command B: `!toc depended-by {file_path}`

**Purpose:** Show all files that the specified file imports from — i.e., the file's forward dependencies ("what do I depend on?"). This is the reverse of `depends-on`.

**Requires:** `{project_root}/.synapse/dep_graph.json` (generated by `!toc_generate`)

### Step B1: Verify dep_graph.json exists

Same as Step A1. If missing, report the same message and stop.

### Step B2: Read the dependency graph

Read `{project_root}/.synapse/dep_graph.json` in full. Parse the `graph` object.

### Step B3: Look up the file

Normalize the provided `{file_path}` (same normalization as Step A3). Look up in `graph`. If not found, report the same not-found message as Step A3.

### Step B4: Retrieve dependencies

Read the file's `imports` array from the graph. If the array is empty, report that this file has no project-internal dependencies.

### Step B5: Enrich with TOC summaries

Read `{project_root}/.synapse/toc.md`. For each file in the `imports` list:
1. Find its entry in the TOC
2. Extract the summary, tags, and exports

If `fingerprints.json` exists, also read it and show each dependency's `purpose` and `key_exports` to give context on what the target file is consuming.

### Step B6: Report results

```markdown
## Depended By: {file_path}

{file_path} imports from {N} file(s):

**1. {filename}**
- **Path:** `{relative_path}`
- **Summary:** {TOC summary}
- **Tags:** {tags from TOC}
- **Purpose:** {purpose from fingerprints, if available}
- **Provides:** {key exports from the dependency that are relevant}

**2. {filename}**
- ...

### Not in TOC
{List any dependency files that appear in dep_graph.json but have no TOC entry — suggest `!toc_update`}
```

## Sub-command C: `!toc cluster {keyword}`

**Purpose:** Find all files related to a keyword by combining keyword matching with dependency graph traversal. Shows the "neighborhood" of a concept — the core files that match plus their direct dependency connections.

**Requires:** `{project_root}/.synapse/toc.md` (for keyword matching) and `{project_root}/.synapse/dep_graph.json` (for dependency expansion). If `dep_graph.json` does not exist, fall back to keyword-only results (same as standard `!toc` search) and suggest running `!toc_generate`.

### Step C1: Read TOC and dependency graph

Read `{project_root}/.synapse/toc.md` in full. Attempt to read `{project_root}/.synapse/dep_graph.json`. If the dependency graph does not exist, note this and proceed with keyword-only matching (skip Steps C3-C4).

### Step C2: Find core files (keyword match)

Search the TOC for entries matching the `{keyword}`. Match against:
- **File names and paths** — the keyword appears in the filename or path segments
- **Summaries** — the keyword appears in the file's description
- **Tags** — the keyword matches a tag

Collect all matching files into a **core set**. These are the "Core files" in the output.

### Step C3: Expand via dependency graph (one level)

For each file in the core set, look up its entry in `dep_graph.json` and collect:
- All files in its `imports` array (files it depends on)
- All files in its `imported_by` array (files that depend on it)

Add these to a **related set**. Remove any files already in the core set from the related set (no duplicates across groups).

### Step C4: Enrich related files

For each file in the related set, look up its TOC entry and extract the summary and tags. If a related file has no TOC entry, include it with a note that it is not documented.

### Step C5: Report results

```markdown
## Cluster: "{keyword}"

### Core Files (matched "{keyword}")
{N} files matched directly:

- **`{path}`** — {summary} [tags: {tags}]
- **`{path}`** — {summary} [tags: {tags}]
- ...

### Related Files (dependency neighbors)
{M} files connected via imports:

- **`{path}`** — {summary} — {relationship: "imported by {core_file}" or "imports {core_file}"}
- **`{path}`** — {summary} — {relationship}
- ...

### Cluster Stats
- **Core files:** {N}
- **Related files:** {M}
- **Total cluster size:** {N+M}
```

If `dep_graph.json` was not available, omit the "Related Files" section and add a note:

```
> Dependency expansion unavailable — `dep_graph.json` not found. Run `!toc_generate` to enable cluster dependency traversal.
> Showing keyword matches only.
```

## Sub-command D: `!toc changes-since {git_ref}`

**Purpose:** Show which TOC-indexed files have changed since a given git reference, and report any purpose, API, or dependency changes detected by comparing current file state against stored fingerprints. This is a **read-only** query — it does NOT modify `fingerprints.json` or `dep_graph.json`. Use `!toc_update` to persist changes.

**Requires:** `{project_root}/.synapse/fingerprints.json` (for comparison). If it does not exist, report a simplified file-change list without semantic analysis.

### Step D1: Get changed files from git

Run:

```bash
git -C {project_root} diff --name-only {git_ref}
```

Collect the list of changed file paths. If the git command fails (bad ref, not a git repo), report the error and stop.

### Step D2: Filter to TOC-indexed files

Read `{project_root}/.synapse/toc.md`. Cross-reference the changed files with the TOC index. Separate into:
- **Indexed files** — changed files that have entries in the TOC
- **Unindexed files** — changed files not in the TOC (mention these briefly but do not analyze them)

### Step D3: Read stored fingerprints

Read `{project_root}/.synapse/fingerprints.json`. For each indexed changed file, check if it has a stored fingerprint.

- **Files with fingerprints** — proceed to Step D4 for semantic comparison
- **Files without fingerprints** — list them as "changed but no stored fingerprint" (no semantic analysis possible)

If `fingerprints.json` does not exist at all, skip semantic analysis entirely and report only the file change list with TOC summaries.

### Step D4: Dispatch scan agents for current fingerprints

Dispatch scan agents (same prompt format as `!toc_generate` Phase 2) for the changed files that have stored fingerprints. Each agent reads the file and reports:
- Purpose (from the fixed category set)
- Key exports (name, kind, params)
- Key imports (project-internal relative paths)
- Complexity rating

**Important:** This is a read-only operation. The scan agents produce current fingerprints for comparison only. Their output is NOT written to `fingerprints.json`.

### Step D5: Compare old vs new fingerprints

For each changed file with both a stored fingerprint and a current scan result, compare:

**Purpose change detection:**
- If `purpose` differs, flag: `PURPOSE CHANGE: {path} changed from {old_purpose} to {new_purpose}`

**API change detection:**
- Compare `key_exports` arrays by name. If more than 50% of export names differ, flag: `API CHANGE: {path} — {N} of {M} exports changed (removed: {list}, added: {list})`
- If an export has the same name but different `kind` or significantly different `params` count, flag: `SIGNATURE CHANGE: {path} — {export_name} changed from {old_kind}({old_params} params) to {new_kind}({new_params} params)`

**Dependency change detection:**
- Compare `key_imports` arrays. If imports were added or removed, flag: `DEPENDENCY CHANGE: {path} — added imports from {list}, removed imports from {list}`

**Complexity change detection:**
- If `complexity` changed (e.g., `simple` to `complex`), note it but do not flag prominently — it is informational.

### Step D6: Report results

```markdown
## Changes Since: {git_ref}

**Total changed files:** {N}
**Indexed in TOC:** {N_indexed}
**With stored fingerprints:** {N_fingerprinted}

### Semantic Changes Detected

#### Purpose Changes
- `{path}` — **{old_purpose}** → **{new_purpose}**
- ...

#### API Changes
- `{path}` — {N} of {M} exports changed (removed: `{list}`, added: `{list}`)
- `{path}` — SIGNATURE CHANGE: `{export_name}` changed from {old} to {new}
- ...

#### Dependency Changes
- `{path}` — added imports from `{list}`, removed imports from `{list}`
- ...

### Changed Files (Full List)

| File | TOC Summary | Semantic Flags |
|---|---|---|
| `{path}` | {summary from TOC} | PURPOSE CHANGE, API CHANGE |
| `{path}` | {summary from TOC} | (no semantic changes) |
| `{path}` | (not in TOC) | — |

### Note
This is a **read-only** analysis. No files were modified.
To persist updated fingerprints and dependency data, run `!toc_update`.
```

If `fingerprints.json` was not available:

```markdown
## Changes Since: {git_ref}

**Total changed files:** {N}

> Semantic analysis unavailable — `fingerprints.json` not found. Run `!toc_generate` to enable fingerprint-based change detection.
> Showing file change list with TOC summaries only.

### Changed Files

| File | TOC Summary |
|---|---|
| `{path}` | {summary from TOC} |
| `{path}` | (not in TOC) |

To persist changes, run `!toc_update`.
```

## Keyword Search (Default)

When the first argument does not match any sub-command (`depends-on`, `depended-by`, `cluster`, `changes-since`), execute the standard keyword search. This is the original `!toc` behavior, preserved unchanged.

### Step 1: Read the Table of Contents

Read `{project_root}/.synapse/toc.md` in full. This is the project's semantic index — it contains summaries, tags, related files, and context for every indexed file.

### Step 2: Identify candidate files

#### Tag-Only Mode

If the query starts with `#` or contains `#`-prefixed terms, extract all tag terms (strip the `#` prefix).

1. Parse the TOC for all entries and their `[tags: ...]` brackets.
2. Filter to entries where ALL specified tags appear in the entry's tag list (case-insensitive).
3. If the query also contains non-tag terms (e.g., `!toc #api user`), further filter by text-matching those terms against filenames and summaries.
4. Tag-filtered results skip the general relevance ranking — they are already narrowed by explicit tag matching. Sort them by directory grouping instead.

Example:
- `!toc #api` — returns all files tagged with `api`
- `!toc #api #backend` — returns files tagged with BOTH `api` AND `backend`
- `!toc #hook explore` — returns files tagged with `hook` whose name or summary matches "explore"

#### General Text Matching

Search the TOC for entries matching the user's query. Match against:

- **File names and paths** — direct match on the filename or path segments
- **Summaries** — semantic match on the description text
- **Tags** — match on any tagged keywords
- **Related files** — match on cross-references

Rank candidates by relevance:
1. **Exact name match** — query matches a filename or export name directly
2. **Summary match** — query terms appear in the file's summary/description
3. **Tag match** — query terms appear in tags
4. **Related file match** — query matches a file referenced by another entry

Select the top 1-5 most relevant candidates.

### Step 3: Verify candidates

For each candidate, **read the actual file** (or the first ~50 lines) to confirm it is genuinely what the user is looking for. This prevents false positives from stale or misleading TOC entries.

If a candidate is NOT what the user wants, drop it and note why.

### Step 4: Report results

When presenting results, also show related files for each match to provide contextual discovery:

#### Related File Traversal

After identifying the top candidates, check their `Related:` entries in the TOC. For each related file:
1. Look up its TOC entry.
2. If it exists, include its path and summary in the result's "Related Context" section.
3. Do NOT recurse further — only show direct (1-hop) related files to avoid information overload.

Only include related files that are themselves documented in the TOC (so their summaries are available). Limit to 3 related files per result to keep output concise. Prioritize related files that are most likely useful — types/interfaces over tests, services over utilities.

Present the results in this format:

```markdown
## TOC Search: "{query}"

### Results

**1. {File name}**
- **Path:** `{path_relative_to_project_root}`
- **Summary:** {1-2 sentence description from TOC + verification}
- **Tags:** {relevant tags}
- **Related Context:**
  - `{related_path_1}` — {summary of that related file, from the TOC}
  - `{related_path_2}` — {summary of that related file, from the TOC}

**2. {File name}**
- ...

### Tag Matches (if tag search)
Found {N} files tagged with #{tag}.

### Not Found (if applicable)
If no relevant files were found, suggest:
- Alternative search terms the user could try
- Which directory might contain what they're looking for
- Whether the file might not be indexed yet (suggest `!toc_update`)
```

## Rules

- **Do not modify any files.** This is a read-only search command. None of the sub-commands write to disk. `changes-since` explicitly does NOT update `fingerprints.json` or `dep_graph.json`.
- **Always verify candidates.** Never report a file based solely on the TOC entry — confirm it by reading the file. (Applies to keyword search and cluster core files. Dependency-based results from `depends-on`, `depended-by`, and cluster related files are reported from the graph without verification since the graph is the source of truth for import relationships.)
- **Be concise.** The user wants to find files fast, not read an essay. Keep summaries to 1-2 sentences.
- **If the TOC is stale or missing entries**, tell the user and suggest running `!toc_update` or `!toc_generate`.
- **If dep_graph.json or fingerprints.json is missing**, tell the user which sub-commands are unavailable and suggest `!toc_generate` to produce them. Do not fail silently — always explain what is missing and why.
- **If the query is ambiguous**, return multiple candidates and let the user decide which is relevant.
- **Prefer precision over recall.** It is better to return 2 correct results than 10 results where 8 are noise.
- **Sub-command parsing is exact.** Only the four defined sub-command names trigger sub-command mode. Partial matches, typos, or similar strings fall through to keyword search.
